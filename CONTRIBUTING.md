# Contributing

Contributions are welcome — improvements to the framework, fixes to examples, or real-world usage patterns you have found helpful.

## What to contribute

- **Clarifications** — if something in the framework is unclear or could be explained better
- **Examples** — worked Impact Plans for different roles (engineer, manager, designer, etc.)
- **Templates** — blank formats for specific contexts
- **Corrections** — anything factually wrong or internally inconsistent

## What not to contribute

- Changes that add complexity without a clear benefit
- Role-specific jargon that makes the framework less universal
- Process overhead that makes the model harder to adopt

## How to contribute

1. Open an issue describing what you want to change and why
2. Fork the repo and make your changes on a branch
3. Open a pull request with a short description of the change

For small fixes (typos, wording), a PR without an issue is fine.
